# ADR-001: レイヤードアーキテクチャの責務分担

## ステータス
メンバー間で合意済み

## コンテキスト
ハンドラ層 → サービス層 → リポジトリ層の 3 層構造を採用する。加えてデータ構造を定義するモデル層を設ける。各層の責務は以下のとおりであり、これを越えた処理を持ち込まない。

| 層 | やること | やらないこと |
|---|---|---|
| ハンドラ | リクエストのバインドとバリデーション、認証情報の抽出、サービス呼び出し、レスポンス変換、ドメインエラーの HTTP ステータスへの変換 | ビジネスロジック、DB の直接操作 |
| サービス | ビジネスロジック、トランザクション管理、複数リポジトリの組み合わせ | HTTP の概念（ステータスコード・ヘッダー等） |
| リポジトリ | DB アクセス（`pgx`）のみ、`pgx` / `pgconn` のエラーをドメインエラーに変換 | ビジネスロジック |
| モデル | データ構造の定義 | ロジック |

ディレクトリ構成（target）:

```
backend/
├── handlers/    # ハンドラ層
├── services/    # サービス層（新設）
├── repositories/  # リポジトリ層（新設）
├── model/       # モデル層
├── helper/      # 共通ヘルパー（リクエストバインド、レスポンス整形など）
└── database/    # 接続プール
```

## 理由
各層の責務を明確にすることで、コードの置き場所の迷いをなくし、変更の影響範囲を限定するため。

## トレードオフ
- 現状はハンドラ層から `pgxpool.Pool` を直接使っている。サービス層・リポジトリ層の分離は段階的に進める
- バインドとバリデーションのヘルパー、ドメインエラーの定義はまだ存在しない。実装時に別途 ADR で方針を定める
